# Execution and Broker Layer

## Purpose

This skill defines how the execution layer for the quant system must be built.

The execution layer is the boundary between the internal decision engine and the external world of real orders, real fills, and real money. Everything upstream — features, signals, risk gates, backtests, paper trades — is hypothetical until an order reaches a broker and a fill comes back.

This layer must be:
- simple and explicit
- resilient to external failures
- deterministic in its internal state management
- paranoid about order integrity
- strictly bounded by risk controls
- observable at every step
- aligned with the same decision logic used in backtesting and paper trading

The execution layer is not where cleverness lives.
It is where discipline becomes real.

---

## When to use

Use this skill whenever you are:

- designing the broker connectivity layer
- implementing order submission and tracking
- handling fill confirmations and reconciliation
- designing partial fill handling
- implementing failover and recovery
- defining execution quality monitoring
- building the bridge between paper trading and live execution
- designing position reconciliation with the broker
- handling connection drops, timeouts, and retries
- defining kill switches and emergency procedures
- deciding what belongs in the execution layer versus the strategy engine
- reviewing whether the system is ready for live deployment

Use this skill especially when there is a risk of:
- treating execution as a simple API call
- ignoring failure modes
- allowing orders without risk validation
- losing track of order state
- diverging from backtested behavior
- building execution logic that cannot be audited

---

## Core mindset

The execution layer exists to translate validated order intents into real market actions with full accountability.

It should not:
- make trading decisions
- override risk gates
- silently retry failed orders without governance
- hide execution failures
- allow untracked state

It should:
- receive validated order intents from the strategy engine
- submit orders to the broker
- track every state transition
- reconcile fills with internal accounting
- detect and handle anomalies
- provide complete audit trails
- fail safely when uncertain

The execution layer is a controlled boundary, not an intelligence layer.

---

## Principles

### 1. The execution layer does not decide; it executes

Order decisions come from the strategy engine after passing through risk gates.

The execution layer receives order intents that have already been:
- generated by signal logic
- validated by risk gates
- sized by position-sizing logic
- approved by exposure checks

The execution layer should never:
- generate its own trades
- modify sizing without authorization
- bypass risk limits
- reinterpret signals
- add discretionary logic

Its job is to faithfully execute what the validated decision pipeline produces.

---

### 2. Order state must be tracked explicitly

Every order must have an explicit lifecycle tracked by a state machine.

Possible states include:
- pending submission
- submitted
- acknowledged by broker
- partially filled
- fully filled
- cancelled
- rejected by broker
- expired
- failed
- unknown or requires reconciliation

State transitions must be:
- logged
- timestamped
- attributable to a specific event
- auditable
- recoverable after restart

Do not represent order state as a collection of booleans or ad hoc flags.

---

### 3. Fills must be reconciled, not assumed

A submitted order is not a filled order.

The system must:
- wait for explicit fill confirmation
- compare expected fill with actual fill
- record fill price, quantity, timestamp, and fees
- update internal position and PnL accounting from confirmed fills only
- detect partial fills and handle them explicitly
- flag discrepancies between expected and actual execution

Never update position state based on order submission alone.

---

### 4. Partial fills are a first-class concern

Real markets produce partial fills.

The system must handle:
- partial fill of the intended quantity
- multiple partial fills over time
- deciding whether to wait, cancel the remainder, or adjust
- accounting for partially filled positions correctly
- risk recalculation after partial fills
- fee accounting on partial fills

A system that assumes all-or-nothing fills will behave incorrectly in production.

---

### 5. Connection failures must be handled, not ignored

Broker connections fail.

The system must define behavior for:
- connection timeout during order submission
- connection drop after submission but before acknowledgment
- connection drop after acknowledgment but before fill
- API rate limiting
- authentication failures
- maintenance windows
- unexpected error responses
- stale or delayed fill reports

For each failure mode, the system must define:
- what is the safe default action?
- should the system retry, wait, or halt?
- how does the system reconcile state after recovery?
- who is notified?

The safe default when state is uncertain should be: do nothing new and alert.

---

### 6. Risk gates must be enforced at the execution boundary

Even though risk validation happens upstream, the execution layer should enforce a final set of hard limits.

These should include:
- max order size
- max position size
- max daily order count
- max daily notional
- max daily loss
- order price sanity checks
- duplicate order detection
- session boundary enforcement
- kill-switch status check

These are not replacements for upstream risk logic. They are last-resort safety barriers.

If the execution layer detects a violation, it must reject the order and alert rather than submit it.

---

### 7. Position reconciliation with the broker is mandatory

The system's internal position state and the broker's reported position must be compared regularly.

Reconciliation should detect:
- position size mismatches
- average price discrepancies
- missing fills
- unexpected positions from manual intervention or external systems
- stale internal state

When a mismatch is detected:
- alert immediately
- do not silently correct internal state without logging
- do not submit new orders until reconciliation is resolved or acknowledged
- preserve both the internal and external views for auditing

Position drift is one of the most dangerous failure modes in live trading.

---

### 8. The execution layer must share core logic with paper trading

The transition from paper trading to live execution should change the execution backend, not the decision logic.

The ideal architecture is:
- same feature pipeline
- same signal logic
- same risk gates
- same order intent generation
- different fill source: simulated fills versus broker fills

This alignment ensures that paper-trading results are meaningful predictors of live behavior.

If the paper-trading and live execution paths diverge significantly, the paper-trading validation becomes unreliable.

---

### 9. Execution quality must be measured

The system should track execution quality over time.

Useful metrics include:
- slippage versus expected fill price
- fill latency
- rejection rate
- partial fill frequency
- spread at time of execution
- execution cost versus backtest assumptions
- order round-trip time
- fill rate
- broker error frequency

These metrics serve two purposes:
- detecting degradation in execution quality
- validating that backtest cost assumptions are realistic

If real execution is consistently worse than backtest assumptions, the backtest is lying.

---

### 10. Emergency procedures must be defined

The system must have clear procedures for:
- kill switch: halt all new orders immediately
- flatten: close all open positions
- disconnect: stop all broker communication
- manual override: allow human intervention with full logging
- recovery: resume from a known safe state

These procedures should be:
- testable in paper-trading mode
- accessible through explicit commands
- logged completely
- not dependent on the strategy engine being healthy

The execution layer must be able to protect capital even if everything else fails.

---

## Broker abstraction principles

### 11. The broker interface should be a thin, typed contract

The system should define a broker trait or interface that abstracts broker-specific details.

The interface should support:
- submit order
- cancel order
- query order status
- query positions
- query account state
- subscribe to fill events
- subscribe to order status events
- health check

Each broker implementation should be a thin adapter behind this contract.

This allows:
- broker replacement without rewriting strategy logic
- simulated broker for paper trading
- mock broker for testing
- multiple broker support later if needed

---

### 12. Broker-specific quirks must not leak into the engine

Each broker has its own:
- order types
- naming conventions
- error codes
- rate limits
- timestamp formats
- fill reporting behavior
- position reporting conventions
- fee structures

These should be handled inside the broker adapter, not spread across the codebase.

The engine should speak in canonical order intents and canonical fill reports.

---

## Order management principles

### 13. Order types should start simple

For v1, support only the order types the strategy actually needs.

Common starting points:
- market order
- limit order
- stop order
- stop-limit order

Do not build support for exotic order types before any strategy requires them.

Each supported order type should have:
- clear semantics
- explicit state machine
- tested lifecycle
- documented broker mapping

---

### 14. Duplicate and conflicting orders must be detected

The system should prevent:
- submitting the same order twice
- submitting a buy while a sell for the same instrument is pending
- exceeding position limits through concurrent orders
- race conditions between signal generation and order confirmation

Use explicit order tracking and pre-submission validation to catch these.

---

## Deployment and operations principles

### 15. The system must support graceful startup and shutdown

On startup:
- reconcile positions with broker
- verify account state
- check kill-switch status
- verify data feed health
- log startup state completely

On shutdown:
- cancel all pending orders (or confirm this is desired)
- reconcile final positions
- log shutdown state completely
- ensure no orders are in flight

Abrupt restarts should trigger reconciliation before any new activity.

---

### 16. Logging must be complete and structured

Every execution event should be logged with:
- timestamp
- order ID
- instrument
- action type
- quantity
- price information
- state transition
- source of the action
- any error or warning

Execution logs are the legal record of what the system did.

They must never be incomplete or ambiguous.

---

### 17. The execution layer should be testable without a real broker

Testing should cover:
- order state machine transitions
- fill reconciliation logic
- partial fill handling
- connection failure scenarios
- risk gate enforcement
- duplicate detection
- position reconciliation
- kill switch behavior
- startup and shutdown sequences

A mock broker adapter should support:
- normal fill behavior
- delayed fills
- partial fills
- rejections
- timeouts
- unexpected state

The execution layer must be as well-tested as the backtesting engine.

## Suggested module split

A reasonable shape for the execution layer:

- `broker/`
  - broker trait definition
  - broker adapters (one per broker)
  - mock broker for testing
  - simulated broker for paper trading

- `orders/`
  - order types
  - order state machine
  - order tracking
  - duplicate detection
  - order validation

- `fills/`
  - fill processing
  - partial fill handling
  - fill reconciliation
  - execution quality metrics

- `positions/`
  - position reconciliation
  - position state
  - broker position comparison

- `safety/`
  - execution-level risk gates
  - kill switch
  - emergency procedures
  - session enforcement

- `monitoring/`
  - execution quality tracking
  - latency metrics
  - error tracking
  - alerting hooks

## Operating instructions

When applying this skill:

1. Never let the execution layer make trading decisions.
2. Track every order through an explicit state machine.
3. Reconcile fills before updating positions.
4. Handle partial fills as a first-class concern.
5. Define behavior for every connection failure mode.
6. Enforce hard risk limits at the execution boundary.
7. Reconcile positions with the broker regularly.
8. Share decision logic with paper trading.
9. Measure execution quality against backtest assumptions.
10. Test the execution layer as rigorously as the backtesting engine.

---

## Guardrails

Never:
- submit orders without passing through risk validation
- assume a submitted order is filled
- ignore partial fills
- hide execution failures
- allow unreconciled position state
- skip duplicate order detection
- let broker-specific details leak into strategy logic
- deploy without kill-switch capability
- treat execution logging as optional
- go live without testing failure scenarios

Never treat the execution layer as "just an API call."

---

## Output style

When using this skill, produce guidance that is:
- safety-conscious
- explicit about failure modes
- paranoid about order integrity
- focused on reconciliation and auditability
- practical about broker realities
- aligned with the paper-trading-to-live transition

Prefer language like:
- "reconcile before trusting"
- "track every state transition"
- "define behavior for this failure mode"
- "enforce risk limits at the boundary"
- "measure execution quality against assumptions"
- "test this with the mock broker"

Avoid language like:
- "just call the API"
- "fills are instant"
- "we can handle errors later"
- "the broker always works"
- "partial fills are rare so ignore them"

---

## Definition of done

An execution layer component should be considered complete only when:

- the order state machine is explicit and tested
- fill reconciliation is implemented
- partial fills are handled
- connection failures have defined behavior
- execution-level risk gates are enforced
- position reconciliation is implemented
- execution quality is measured
- the component works with both mock and real broker adapters
- emergency procedures are testable
- logging is complete and structured

---

## Final principle

A strong execution layer is not the one that submits orders fastest.

It is the one that never loses track of what it did, what happened, and what went wrong.
